üöÄ Starter Activity: Code Duplication Problem

The Problem

Look at these two classes from our RPG game:

class Player: def __init__(self, name): self.__name = name self.__health = 100 self.__max_health = 100 self.__level = 1 def get_name(self): return self.__name def take_damage(self, amount): self.__health -= amount if self.__health < 0: self.__health = 0
class Enemy: def __init__(self, name): self.__name = name self.__health = 50 self.__max_health = 50 self.__damage = 10 def get_name(self): return self.__name def take_damage(self, amount): self.__health -= amount if self.__health < 0: self.__health = 0
‚ö†Ô∏è
Spot the Problem: How much code is duplicated between these classes?
  • Both have __name, __health, __max_health
  • Both have identical get_name() methods
  • Both have identical take_damage() methods

This violates the DRY principle (Don't Repeat Yourself)!

üß¨ What is Inheritance?

üí°
Definition: Inheritance is a mechanism where a new class (child/derived/subclass) is created based on an existing class (parent/base/superclass), inheriting its attributes and methods.

The Family Tree Analogy

Inheritance works like a family tree:

  • üë¥ Parent class: Has characteristics that can be passed down
  • üë∂ Child class: Inherits characteristics from parent
  • ‚ûï Extension: Child can add its own unique characteristics
  • üîÑ Modification: Child can modify inherited behaviors

Benefits of Inheritance

‚ôªÔ∏è Code Reuse

Write common code once in the parent class

üèóÔ∏è Logical Structure

Models real-world relationships naturally

üîß Easier Maintenance

Changes in parent automatically apply to children

üìà Extensibility

Easy to add new types without starting from scratch

üîó The "Is-A" Relationship

üí°
Key Concept: Inheritance represents an "is-a" relationship. If class B inherits from class A, then B "is a" type of A.

Valid "Is-A" Relationships

Test these relationships - which ones make sense?

  • ‚úÖ A Player is a Character
  • ‚úÖ An Enemy is a Character
  • ‚úÖ A Wizard is a Player
  • ‚ùå A Weapon is a Character (doesn't make sense!)
  • ‚ùå A Character is a Player (wrong direction!)

Is-A vs Has-A

üß¨ Is-A (Inheritance)

Use when one thing is a type of another

  • Dog is-a Animal ‚úÖ
  • Car is-a Vehicle ‚úÖ
  • Warrior is-a Player ‚úÖ

üì¶ Has-A (Composition)

Use when one thing contains another

  • Player has-a Weapon ‚úÖ
  • Car has-a Engine ‚úÖ
  • Character has-a Inventory ‚úÖ

üèóÔ∏è Base Classes and Derived Classes

Terminology

Different names for the same concepts:

  • Base Class = Parent Class = Superclass
  • Derived Class = Child Class = Subclass

What Gets Inherited?

‚úÖ Inherited

  • All public methods
  • All public attributes
  • Protected members (single _)

‚ùå Not Inherited

  • Private attributes (__)
  • Private methods (__)
  • Constructor (but can be called)

üé® Designing the Base Class

Character Base Class Design

Character

Attributes (Common to all characters):
  • __name: string
  • __health: integer
  • __max_health: integer
  • __x_position: integer
  • __y_position: integer
Methods (Common behaviors):
  • get_name()
  • get_health()
  • take_damage(amount)
  • heal(amount)
  • move(x, y)
  • is_alive()
üí°
Design Tip: Put in the base class only what is truly common to ALL derived classes. Specific features go in the derived classes.

üìù Inheritance Syntax

Python Syntax

# Base class definition class Character: def __init__(self, name, health): self.__name = name self.__health = health self.__max_health = health def get_name(self): return self.__name def take_damage(self, amount): self.__health = max(0, self.__health - amount) # Derived class - inherits from Character class Player(Character): # Note the parentheses! def __init__(self, name): # Call parent constructor super().__init__(name, 100) # Players start with 100 health self.__level = 1 self.__experience = 0

Pseudocode Syntax (Cambridge 9618)

CLASS Character PRIVATE Name : STRING PRIVATE Health : INTEGER PRIVATE MaxHealth : INTEGER PUBLIC PROCEDURE New(N : STRING, H : INTEGER) Name ‚Üê N Health ‚Üê H MaxHealth ‚Üê H ENDPROCEDURE PUBLIC FUNCTION GetName() RETURNS STRING RETURN Name ENDFUNCTION ENDCLASS CLASS Player INHERITS Character // Note the INHERITS keyword PRIVATE Level : INTEGER PRIVATE Experience : INTEGER PUBLIC PROCEDURE New(N : STRING) SUPER.New(N, 100) // Call parent constructor Level ‚Üê 1 Experience ‚Üê 0 ENDPROCEDURE ENDCLASS

ü¶∏ The super() Function

üí°
Purpose: super() gives you access to methods in the parent class. Most commonly used to call the parent's constructor.

Why Use super()?

  • ‚úÖ Avoid duplicating parent class initialization code
  • ‚úÖ Ensure parent class is properly set up
  • ‚úÖ Maintain the inheritance chain
  • ‚úÖ Make code more maintainable

Common super() Patterns

class Player(Character): def __init__(self, name, player_class): # Pattern 1: Call parent constructor first super().__init__(name, 100) # Then add child-specific attributes self.__player_class = player_class self.__mana = 50 self.__inventory = [] def take_damage(self, amount): # Pattern 2: Extend parent method print(f"{self.get_name()} takes {amount} damage!") super().take_damage(amount) # Call parent's version # Add child-specific behavior if self.get_health() == 0: print("Game Over!")

‚ö†Ô∏è Common super() Mistakes

‚ùå Wrong

class Player(Character): def __init__(self, name): # Forgot to call super()! self.__level = 1

Parent's __init__ never runs!

‚úÖ Correct

class Player(Character): def __init__(self, name): super().__init__(name, 100) self.__level = 1

Parent is properly initialized!

‚ûï Extending Functionality

Child classes can add new attributes and methods that don't exist in the parent:

Adding New Attributes

class Player(Character): def __init__(self, name): super().__init__(name, 100) # New attributes specific to Player self.__level = 1 self.__experience = 0 self.__gold = 50 self.__inventory = []

Adding New Methods

class Player(Character): # ... previous code ... # New method only for Players def gain_experience(self, amount): self.__experience += amount # Check for level up if self.__experience >= 100 * self.__level: self.level_up() def level_up(self): self.__level += 1 self.__experience = 0 print(f"{self.get_name()} reached level {self.__level}!") def add_to_inventory(self, item): self.__inventory.append(item) print(f"{item} added to inventory")

üîÑ Method Overriding

Child classes can also override (replace) parent methods:

class Boss(Enemy): def __init__(self, name, damage): super().__init__(name, damage) self.__phase = 1 # Override parent's take_damage method def take_damage(self, amount): # Bosses take reduced damage reduced_damage = amount // 2 super().take_damage(reduced_damage) # Boss-specific behavior if self.get_health() < 25 and self.__phase == 1: self.__phase = 2 print(f"{self.get_name()} enters rage mode!")
‚ö†Ô∏è
Note: When overriding methods, you can choose to:
  • Completely replace the parent's behavior
  • Call super() to keep parent behavior and add to it
  • Call super() with modified parameters

üéÆ Refactoring Our RPG with Inheritance

Let's refactor our RPG classes to use inheritance properly:

Step 1: Create the Base Character Class

class Character: def __init__(self, name, health): self.__name = name self.__health = health self.__max_health = health self.__x = 0 self.__y = 0 self.__is_alive = True # Getters def get_name(self): return self.__name def get_health(self): return self.__health def get_position(self): return (self.__x, self.__y) def is_alive(self): return self.__is_alive # Common methods def take_damage(self, amount): if amount > 0: self.__health = max(0, self.__health - amount) if self.__health == 0: self.__is_alive = False def heal(self, amount): if amount > 0 and self.__is_alive: self.__health = min(self.__max_health, self.__health + amount) def move(self, dx, dy): self.__x += dx self.__y += dy

Step 2: Create the Player Subclass

class Player(Character): def __init__(self, name, player_class="Warrior"): # Players start with 100 health super().__init__(name, 100) # Player-specific attributes self.__player_class = player_class self.__level = 1 self.__experience = 0 self.__gold = 50 self.__inventory = [] self.__skills = self.__initialize_skills() def __initialize_skills(self): if self.__player_class == "Warrior": return ["Slash", "Block"] elif self.__player_class == "Mage": return ["Fireball", "Heal"] else: return ["Attack"] # Player-specific methods def gain_experience(self, amount): self.__experience += amount while self.__experience >= 100 * self.__level: self.__experience -= 100 * self.__level self.__level += 1 print(f"{self.get_name()} reached level {self.__level}!") def add_gold(self, amount): if amount > 0: self.__gold += amount def spend_gold(self, amount): if 0 < amount <= self.__gold: self.__gold -= amount return True return False

Step 3: Create the Enemy Subclass

class Enemy(Character): def __init__(self, name, enemy_type, health, damage): super().__init__(name, health) # Enemy-specific attributes self.__enemy_type = enemy_type self.__damage = damage self.__loot_table = self.__initialize_loot() def __initialize_loot(self): if self.__enemy_type == "Goblin": return {"gold": 10, "experience": 25} elif self.__enemy_type == "Orc": return {"gold": 25, "experience": 50} else: return {"gold": 5, "experience": 10} def get_damage(self): return self.__damage def get_loot(self): return self.__loot_table.copy() def attack(self, target): if self.is_alive(): print(f"{self.get_name()} attacks {target.get_name()} for {self.__damage} damage!") target.take_damage(self.__damage)

üß™ Testing Our Inheritance Structure

Let's test our refactored classes:

# Create instances player = Player("Hero", "Warrior") goblin = Enemy("Goblin Scout", "Goblin", 30, 5) orc = Enemy("Orc Warrior", "Orc", 50, 12) # All characters can use inherited methods print(player.get_name()) # "Hero" - inherited method print(goblin.get_health()) # 30 - inherited method player.move(5, 3) # inherited method # Player-specific methods player.gain_experience(50) # only Player has this # Enemy-specific methods goblin.attack(player) # only Enemy has this

‚ö†Ô∏è Common Inheritance Mistakes

1. Forgetting to Call super().__init__()

‚ùå Wrong

class Player(Character): def __init__(self, name): self.__level = 1 # Parent's __init__ never called! # __name, __health not initialized!

‚úÖ Correct

class Player(Character): def __init__(self, name): super().__init__(name, 100) self.__level = 1

2. Wrong Inheritance Relationships

‚ùå
Common Mistake: Using inheritance for "has-a" relationships
# WRONG: A Player is not a type of Weapon! class Player(Weapon): # This makes no sense! pass

3. Accessing Private Parent Attributes

‚ùå This Won't Work

class Enemy(Character): def display_info(self): # Can't access parent's private attributes! print(self.__name) # AttributeError!

‚úÖ Use Parent's Public Methods Instead

class Enemy(Character): def display_info(self): # Use the getter method print(self.get_name()) # Works!

4. Circular Inheritance

‚ùå
Never do this:
# This creates an infinite loop! class A(B): pass class B(A): pass
