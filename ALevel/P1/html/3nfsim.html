A table is in 3NF if it is in 2NF and contains no transitive dependencies.

A transitive dependency is when a non-key attribute depends on another non-key attribute, not on the primary key. Let's explore this with an example.

Identifying the Transitive Dependency

The problem is a transitive dependency, which can lead to data anomalies.

Customer_Orders Table

OrderID CustomerID CustomerName ManagerName
101C1BookEnds LtdJohn Smith
102C2The PaperbackJane Doe
103C1BookEnds LtdJohn Smith
104C3Readers CornerPeter Jones

The dependency path is: OrderID → CustomerID → ManagerName.
Since ManagerName depends on a non-key attribute (CustomerID), it's a transitive dependency.

The Solution: Decomposing the Table

We fix this by splitting the table, creating a new one for the attributes in the transitive dependency.

Key Takeaways:

  • 3NF requires a table to first be in 2NF.
  • 3NF eliminates transitive dependencies.
  • This is done by decomposing tables to separate the transitive dependencies.
  • 3NF reduces redundancy and prevents data anomalies.

Transitive Dependency Alert!

ManagerName is determined by CustomerID, not the primary key OrderID. This is a classic transitive dependency. If the manager for a customer changes, you'd have to update multiple rows, risking data inconsistency. This violates 3NF.
